# Precise Git Commit Skill

生成遵循 Conventional Commits 规范、结构清晰且**高度真实（绝不乱编/夸大）**的 Git 提交信息说明。

## 触发条件

当用户发出明确的提交指令时触发，例如：“帮我生成 commit”、“写一下提交信息”、“提交代码”、“帮我 commit”。

## 核心原则（严禁乱编）

1. **事实唯一性**：提交信息必须 100% 来源于 `git diff` 和 `git status` 的真实代码改动。
2. **拒绝幻觉与夸大**：如果只改了配置文件或删除了冗余样式，绝不能在 commit 中虚构“实现了 OSS 云存储上传”或“重构了核心切块算法”等未发生的事情。
3. **精确归类**：准确区分 `feat` (新功能)、`fix` (修复)、`refactor` (重构)、`chore` (杂项/配置/清理)、`docs` (文档)。

## 执行步骤

### 1. 深度盘点改动

运行 terminal 命令审查工作区：

- 执行 `git status` 查看变更文件列表。
- 执行 `git diff --staged`（若暂存区为空则执行 `git diff`）逐行阅读具体的代码增删内容。

### 2. 生成规范详细的 Commit Message

按照以下标准格式组织内容（区分中英文，通常默认根据项目语境或用户习惯生成简明清晰的结构）：

```text
<type>(<scope>): <简明扼要的标题，不超过 50 字>

<详细描述背景或目的，可选>

变更细节：
- [后端/引擎] 具体的真实改动点 1
- [前端/看板] 具体的真实改动点 2
- [配置/工具] 具体的真实改动点 3
```

**Type 规范列表**：

- `feat`: 新增面向用户的特性或核心功能接口
- `fix`: 修复 Bug 或逻辑缺陷
- `refactor`: 代码重构（不改变外部行为）
- `chore`: 构建配置、依赖调整、冗余文件清理、拼写检查字典更新等
- `docs`: 仅文档或注释变更

### 3. 向用户交付或提交

1. 将生成好的 Commit Message 以 Markdown 代码块形式向用户展示，并一一对应说明改动了哪里。
2. 询问用户：“提交信息如上，是否需要我直接为您执行 `git commit`？”（若用户发起请求时已明确要求直接提交，则直接执行 `git commit -m "<message>"` 并反馈结果）。
